| --- | --- | --- | --- | --- | --- |
| `DATA_CONNECTOR -> source projection` | `source`, `output_name`, `columns[]` | `columns` comes from selected `data_source.yaml` schema; allow `all_columns=true` for exploratory runs | Dataset with selected source columns only | Omitted source columns are unavailable downstream unless re-fetched | Use before pulling rows when the user knows which source columns are needed for signals, summaries, reports, or joins. Example: fetch only `order_id`, `trader_id`, `order_time`, `side`, `quantity`, `status`. |
| `DATA_CONNECTOR -> source filters` | `source`, `output_name`, `filters[]` | Each filter shows `column`, `operator`, `value`; columns come from `data_source.yaml`; values can be literal or `{context.key}`; operators include `eq`, `neq`, `gt`, `gte`, `lt`, `lte`, `between`, `in`, `contains` | Dataset containing only rows matching source-native filters | Filter audit metadata / resolved query | Use before pulling rows whenever condition uses physical source columns. | `DATA_CONNECTOR -> source sort` | `source`, `output_name`, `sort[]` | Each sort row shows `column`, `direction`; columns come from source schema | Dataset ordered by source-native fields | Stable ordering for reports/summaries | Use before pulling when source can sort efficiently. | `DATA_CONNECTOR -> source limit` | `source`, `output_name`, `limit` | Optional `offset` / pagination token for large sources | Dataset capped at source | Count metadata may show returned count vs total available if connector supports it | Use before pulling to avoid huge payloads. | `DATA_CONNECTOR -> source query bundle` | `source`, `columns[]`, `filters[]`, `sort[]`, `limit`, `output_name` | If `filter_mode=builder` show structured query controls; if `filter_mode=raw_query` show raw query and still allow `columns/sort/limit` when supported | Minimal dataset needed by workflow | Resolved query / query plan metadata | Preferred default for connectors. Pull less data first, then use downstream transforms only for values created after the pull. |
| `FILTER_ROWS` | `input_name`, `condition`, `output_name` | Condition builder uses columns available after upstream transforms/signals/joins; optional raw expression advanced mode | Filtered DataFrame | Filtered row count metadata | Use after pull only when condition depends on derived columns. Example: keep `_signal_flag == true`, `signed_notional > 10000000`, `_highlight_label != ''`. Do not use for simple source filters that can be pushed into connector. |
| `SELECT_COLUMNS` / `PROJECT_SCHEMA` | `input_name`, `columns[]`, `output_name` | Columns come from upstream lineage, including derived/signal/highlight columns; optional rename mapping | Projected DataFrame | Column mapping metadata | Use after enrichment when preparing report tabs, CSV exports, LLM dataset summaries, or join inputs. Example: keep `exec_id`, `exec_time`, `side`, `signed_notional`, `_signal_score`, `_signal_reason`. |
| `SORT_LIMIT` | `input_name`, `sort[]`, `limit`, `output_name` | Sort columns come from upstream lineage; can sort on derived/signal fields | Sorted/capped DataFrame | Returned row count metadata | Use after scoring/enrichment for top-N evidence. Example: top 25 rows by `_signal_score desc`, latest 50 communications by `timestamp desc`, largest 100 trades by `abs_notional desc`. |
